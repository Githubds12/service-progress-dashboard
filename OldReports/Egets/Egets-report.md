## 1. Executive Summary
E-GetS (com.egets.takeaways) implements a multi-step authentication flow involving device registration, SMS triggering, and encrypted OTP submission. The application uses a custom security protocol with payload encryption (E1) and request signing (S2). All sensitive API interactions are protected by unique device identifiers, timestamps, and cryptographic signatures. Automation feasibility is medium, requiring the reverse engineering of the custom E1 encryption and S2 signing logic to generate valid request payloads.

## 4. Security & Reversing Notes

### Encryption/Signing Mechanisms

**1. Device Fingerprinting**
- **Header**: `x-device-uuid`
- **Example**: `8564b63f-ffd2-465a-aebd-f2466dcafa35`
- **Analysis**: A static or semi-persistent UUID generated by the Android application to track the device.

**2. Request Signing (S2)**
- **Header**: `x-sign`
- **Purpose**: Protects against request tampering.
- **Analysis**: The signature is a Base64 encoded string generated from the request parameters, headers, and a client-side secret. It changes with every request and is tied to the timestamp.

**3. Payload Encryption (E1)**
- **Field**: `data`
- **Analysis**: Sensitive information, including the phone number and OTP, is encrypted using the E1 protocol before being sent in the `data` field. The encryption likely uses AES or a similar symmetric cipher.

**4. SMS Verification Token**
- **Parameter**: `verify`
- **Example**: `1777451042-a9h5CxgMyGbq9CwuWor7dTe+d25rv/UPda6Ch65r2iU=`
- **Analysis**: A dynamic token required for the `/captcha/send` endpoint. It consists of a timestamp and a hash/HMAC, ensuring that SMS requests are tied to a specific app-initialized session.

## 5. Conclusion

### Automation Feasibility: 55%

### Critical Blockers:
1. **Custom E1 Encryption**: The registration and number submission payloads are encrypted. Automation requires implementing the exact E1 encryption routine found in the APK.
2. **S2 Signature Generation**: Every request must be signed. Reversing the signing algorithm (likely HMAC or RSA) is mandatory.
3. **Dynamic Verify Token**: The SMS trigger requires a `verify` token that is generated client-side, making simple replays impossible without the generation logic.
4. **Device/Session Consistency**: The backend likely validates the consistency between the UUID, request ID, and signatures.
